# NZ Police Learning Module Proofreader

## Overview

This skill reviews NZ Police Detective Development Programme learning modules for:
- **Proofreading** — grammar, spelling, punctuation, and phrasing
- **Consistency** — terminology, tense, and style used uniformly throughout
- **Structure** — sections appear in the correct order and follow the established module template
- **Content logic** — explanations make sense, examples align with the law described, learning objectives match the content
- **Formatting** — headings, tables, bullet lists, and legislation boxes follow the established style

The output is a single Word (.docx) file containing:
1. **Word comments** — categorised annotations explaining each issue found
2. **Tracked changes** — inline suggested rewrites the author can Accept or Reject

---

## Established Module Structure

Every module must follow this structure in this order. Flag any missing, misplaced, or extra sections.

```
1.  COVER PAGE
    - Programme name ("Detective Development Programme [YEAR]")
    - White text on coloured background image
    - NOTE: The background image will vary between modules — do NOT flag a different cover image as an issue

2.  COPYRIGHT PAGE
    - "© New Zealand Police [YEAR]"
    - "First published by"
    - "Professional Development"
    - "The Royal New Zealand Police College"
    - "Edited & Published [DATE]"
    - Study guide disclaimer paragraph
    - Reproduction restriction paragraph
    - Legislation currency warning paragraph

3.  USING THIS MODULE
    - Welcome paragraph
    - Overview of what the module covers
    - Part 1 / Part 2 explanation (DDC vs DMP)
    - Numbered list: Part 1 description, Part 2 description
    - Note about online course and interactive activities
    - Assessment explanation referencing Learning Objectives
    - Closing encouragement paragraph

4.  TABLE OF CONTENTS

5.  PART ONE: DDC LEARNING OBJECTIVES
    - Welcome paragraph for Part One
    - Objective statement
    - "You will be assessed on your knowledge and understanding of these topics:"
    - Offences (bulleted list)
    - Case law (bulleted list)
    - Case law - Universal (bulleted list)
    - Assessment application statement

6.  [OFFENCE NAME] – LEGISLATION
    - Legislation in a two-column bordered table
      - Left column header: "LEGISLATION"
      - Right column: full statutory text, verbatim, with subsections

7.  [OFFENCE NAME] – ELEMENTS
    - One sub-section per charge variant (e.g., 267(1)(a), 267(1)(b), 267(1)(c))
    - Each sub-section: section reference as H2, then bulleted element list

8.  INTRODUCTION (optional bridging section)

9.  UNDERSTANDING [OFFENCE NAME]
    - Actus reus explanation
    - Mens rea explanation
    - Subsections for each key concept (e.g., Intentionally, Recklessly, Damage, Fire, Property)
    - "Example:" callouts where appropriate
    - Case law references in context

10. THE SPECIFIC MENS REA FOR [OFFENCE NAME]
    - One sub-section per charge variant requiring specific intent
    - Case law references with block quotes or paraphrasing
    - "Example:" callouts where appropriate

11. PART TWO: DMP LEARNING OBJECTIVES (mirrors Part One structure)
    - DMP offences list
    - DMP case law list
    - DMP case law - Universal list

12. [ADDITIONAL OFFENCES] — each following the pattern:
    - Legislation section (table)
    - Elements section (bullets)
    - Understanding section (prose + examples)

13. SUPPLEMENTARY SECTIONS (e.g., Fire Safety FAQs)
    - FAQ format: bold question, prose answer, bulleted sub-points with mitigation
```

## Style Rules (from the Arson Module reference)

### Language — New Zealand English
All modules are written in **New Zealand English**. Apply the rules below when proofreading — flag deviations as `[LANGUAGE]` issues.

#### Spelling and word forms

| NZ spelling (correct) | Flag as error |
|---|---|
| programme (general) | program — *unless computing context* |
| offence, defence | offense, defense |
| licence (noun) / license (verb) | license for both |
| organise, recognise, authorise, standardise | -ize endings |
| analyse, paralyse | analyze, paralyze |
| behaviour, neighbour, honour, colour, favour, labour | -or endings |
| centre, metre (unit of length) | center, meter — *flag "meter" unless it means a measuring device* |
| travelled, travelling, labelled, labelling | single-l forms |
| practise (verb) / practice (noun) | practice for both |
| judgement (general use) | judgment — *retain only in explicitly legal/court contexts* |
| tyre, storey (building level) | tire, story |
| whilst | while — *acceptable; flag only if inconsistent with rest of document* |
| cheque | check — *flag in financial contexts* |

#### Dates, times, numbers, and units

- **Dates:** 13 March 2026 — not 13/03/26 or March 13, 2026
- **Times:** 5 pm, 10.30 am — not AM/PM in caps, not 5:00 PM
- **Numbers:** spell out one to nine; use numerals for 10 and above — never start a sentence with a numeral
- **Measurements:** metric with a space before unit: 5 km, 20 °C — not "5kms", "5kph"; use km/h not kph
- **Thousands separator:** 1,000 — not 1000; decimals use a point: 3.5

#### Punctuation and quotation marks

- **Primary quotations:** single quotation marks — 'like this'
- **Quotes within quotes:** double quotation marks — 'He said "guilty" aloud'
- **Oxford comma:** use only if omitting it creates genuine ambiguity — do not add it everywhere
- **Dashes:** en dash (–) for ranges with no spaces (2019–2022); em dash (—) for a break in thought; flag inconsistent dash usage within the document

#### Capitalisation and terminology

- Capitalise official names and proper nouns; use lower case for generic references
  - "New Zealand Police" (institution) vs "the police" (generic)
  - "Sergeant Smith" (rank used with name) vs "the sergeant" (generic reference)
- "Crown" (not "State" or "Government") when referring to the prosecution
- "Police" (capitalised, no article) when referring to New Zealand Police as an institution
- Legislation references use NZ statute names and years exactly as enacted
- Expand acronyms on first use, then use the acronym — e.g., Royal New Zealand Police College (RNZPC)

#### Te Reo Māori

- Use correct macrons throughout: Māori, whānau, tamariki, Aotearoa — flag any missing macrons
- **Do not italicise** Māori words; treat them as standard NZ English
- Use preferred bilingual forms where relevant: "Aotearoa New Zealand" per house style

#### Tone and plain language

- Active voice is preferred — flag passive constructions where active would be clearer
- Short sentences and concrete verbs — flag unnecessary nominalisations (e.g., "make a decision" → "decide")
- Remove redundancy and jargon; flag unexplained technical terms
- Maintain respectful, inclusive language throughout

### Headings
- **H1 (Heading 1):** ALL CAPS, dark navy/dark blue colour (e.g., `1F3864`), used for major section titles
- **H2 (Heading 2):** Title Case, used for sub-sections within a major section
- **H3 (Heading 3):** Title Case or sentence case, used for sub-sub-sections
- Section numbers are NOT used in headings (headings are descriptive text only)

### Legislation Tables
- Always presented in a two-column table
- Left column: label "LEGISLATION" (bold, centred, dark background or border)
- Right column: verbatim statutory text, preserving all subsection lettering
- Table has visible borders
- Never paraphrase legislation — it must be reproduced exactly

### Elements Sections
- Each element on a separate bullet point
- Elements should exactly match the statutory wording
- No additional explanation in the elements list — explanation goes in "Understanding" sections

### Body Text
- Font: Calibri or the document's defined body font — do NOT mix fonts
- Size: 11pt body text (or as established in the reference document)
- Line spacing: consistent throughout (1.15 or as set in styles)
- Paragraphs separated by spacing, not blank lines

### Lists
- Use Word's built-in list styles (not manual hyphens or unicode bullets)
- Nested lists used only when structurally necessary (e.g., mitigation under a hazard)
- Numbered lists for sequential steps; bulleted lists for non-sequential items

### Terminology Consistency
These terms must be used consistently throughout. Flag any variation:
- "actus reus" (not "actus reous" or "AR")
- "mens rea" (not "MR")
- "Detective Development Course" / "DDC"
- "Detective Modular Programme" / "DMP"
- "Detective Development Programme" (the overarching programme — not "Detective Development Program")
- "Crimes Act 1961" (not "Crimes Act" alone on first reference)
- Section references: "Section 267(1)(a), Crimes Act 1961" — use full reference on first use per section; abbreviated form (s267(1)(a)) acceptable in elements lists only
- "claim of right" (lower case)
- "immovable property" (lower case)
- Case law citations: "[Name] v [Name] [year] [court] [number]" — check for consistent formatting

### Tense and Voice
- Present tense for describing the law ("The section requires...", "The prosecution must prove...")
- Past tense for case facts ("The defendant argued...", "The Court held...")
- Active voice preferred

### Example Callouts
- The word "Example:" should appear as a standalone label (bold or styled) before the example text
- Examples must directly illustrate the point made in the preceding paragraph

### Learning Objectives
- Must be phrased as testable outcomes
- Must match the content that follows in the module body
- Any offence or case law listed in Learning Objectives must appear in the body of the module

## Review Process

When the user uploads a module to review, follow these steps:

### Step 1 — Read the DOCX skill
Read `/mnt/skills/public/docx/SKILL.md` before doing any file work.

### Step 2 — Extract and analyse the uploaded module
```bash
pandoc [uploaded_file] -t plain --wrap=none > module_plain.txt
python scripts/office/unpack.py [uploaded_file] unpacked_module/
```
Read the plain text carefully from top to bottom.

### Step 3 — Read any reference documents
If the user has uploaded reference documents (previous approved modules, style guides, glossaries), extract and read them the same way. Use them to:
- Verify terminology accuracy
- Confirm correct legislative wording
- Check that case law citation format is consistent (no external database verification is required)

### Step 4 — Conduct the review
Work through the document systematically, cataloguing every issue found under these categories:

| Category tag | What it covers |
|---|---|
| `[STRUCTURE]` | Missing, misplaced, or extra sections |
| `[GRAMMAR]` | Grammar, spelling, punctuation errors |
| `[LANGUAGE]` | NZ English conventions: spelling variants, dates, numbers, punctuation style, macrons, Te Reo, capitalisation |
| `[CONSISTENCY]` | Same concept referred to with different terms |
| `[CONTENT]` | Logical errors, incorrect law, unsupported claims |
| `[FORMATTING]` | Heading level wrong, list style wrong, table format wrong |
| `[LEARNING OBJ]` | Mismatch between learning objectives and body content |
| `[LEGISLATION]` | Verbatim statutory text differs from the Act |

### Step 5 — Unpack the original document and add comments + tracked changes
```bash
python scripts/office/unpack.py [uploaded_file] unpacked_review/
```

For each issue found:
1. Add a **Word comment** using `comment.py` with the category tag, a clear explanation of the issue, and the reason it matters
2. Add a **tracked change** (insertion + deletion) showing the suggested fix

Author name for all changes and comments: **"Reviewer"**

```bash
python scripts/comment.py unpacked_review/ [id] "[CATEGORY] Explanation of issue and why it matters."
```

Then add the corresponding tracked change markers in `document.xml`.

### Step 6 — Pack and deliver
```bash
python scripts/office/pack.py unpacked_review/ /mnt/user-data/outputs/[ModuleName]_reviewed.docx --original [uploaded_file]
```

Validate the output. Then present the file to the user and give a brief summary covering:
- Total number of issues found
- Breakdown by category
- Any critical issues (structural problems or incorrect legislation) called out specifically

## Comment Format

Each comment must follow this format:
```
[CATEGORY] Issue description. | Suggestion: What to do to fix it.
```

Example:
```
[CONSISTENCY] "Detective Development Program" used here but "Detective Development Programme" used elsewhere (page 3). | Suggestion: Change to "Detective Development Programme" throughout — the correct NZ spelling uses "Programme".
```

```
[STRUCTURE] The "Using This Module" section is missing the closing encouragement paragraph that appears in all modules. | Suggestion: Add a short closing paragraph (e.g., "Best of luck on the Detective Development Programme. Remember, every Detective has started where you are now.").
```

```
[LEGISLATION] The text of s267(1)(a) omits "or recklessly" from "intentionally or recklessly damages". | Suggestion: Restore the verbatim statutory wording: "intentionally or recklessly damages by fire or by means of any explosive any property if he or she knows or ought to know that danger to life is likely to ensue".
```
